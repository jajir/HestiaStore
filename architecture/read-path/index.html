    üìñ Read Path

    This page explains how reads resolve values with low latency and predictable I/O. It walks through point lookups, range iteration, and the interplay of caches, Bloom filter, and the sparse index, mapped to concrete classes in the codebase.

    üß≠ High‚ÄëLevel Flow (Point Lookup)

    1. API call: Index.get(key)
    2. Check the index‚Äëlevel unique buffer (latest in‚Äëprocess writes)
    3. Locate the target segment using the key‚Üísegment map
    4. Inside the segment: consult delta cache ‚Üí Bloom filter ‚Üí sparse index ‚Üí local scan

    Lookups are read‚Äëafter‚Äëwrite consistent thanks to the in‚Äëmemory buffers.

    üö™ Entry Point and First‚ÄëLevel Cache

    • sst/SstIndexImpl#get(K) does:
    • Check the index‚Äëlevel UniqueCache (holds latest writes prior to flush)
    • If miss, find SegmentId via KeySegmentCache.findSegmentId(key)
    • Delegate to Segment.get(key)

    Key classes: sst/SstIndexImpl.java, sst/KeySegmentCache.java, cache/UniqueCache.java.

    Behavior

    • Cache hit and non‚Äëtombstone ‚Üí return value
    • Cache hit and tombstone ‚Üí treat as absent
    • Otherwise fall back to the segment path below

    üß© Per‚ÄëSegment Read Path

    segment/SegmentImpl#get(key) uses SegmentSearcher with lazily loaded segment data:

    1. Delta cache probe: in‚Äëmemory map of the segment‚Äôs pending updates (merged from delta files). If hit and value not a tombstone ‚Üí return; tombstone ‚Üí absent.
    2. Bloom filter: bloomFilter.isNotStored(key) guards the on‚Äëdisk path. If ‚Äúnot stored‚Äù ‚Üí absent.
    3. Sparse index ("scarce index"): returns a chunk start position for keys ‚â• query.
    4. Local scan: within at most N keys (maxNumberOfKeysInIndexPage) starting at that chunk, compare keys in ascending order and stop as soon as the target is found or passed.
    5. If the sparse index pointed us into the file but no exact key was found, mark a false positive on the Bloom filter for metrics and return absent.

    Key classes: segment/SegmentSearcher.java, segment/SegmentIndexSearcher.java, scarceindex/ScarceIndex.java, bloomfilter/BloomFilter.java.

    üîÅ Range Scans and Full Iteration

    • Index.getStream() and Index.openSegmentIterator(...) produce iterators over all data:
    • sst/SegmentsIterator chains Segment.openIterator() across all segments in order.
    • segment/SegmentImpl.openIterator() merges the on‚Äëdisk main SST with the segment‚Äôs delta cache via MergeDeltaCacheWithIndexIterator, skipping tombstones.
    • The per‚Äësegment iterator is wrapped with EntryIteratorWithLock using an OptimisticLock. If a write changes the segment version mid‚Äëscan, the iterator stops gracefully (no partial records).
    • At the top level, EntryIteratorRefreshedFromCache overlays the index‚Äëlevel unique buffer so that the iterator sees the latest writes even before they‚Äôre flushed to disk.

    Key classes: sst/SegmentsIterator.java, segment/MergeDeltaCacheWithIndexIterator.java, sst/EntryIteratorRefreshedFromCache.java, EntryIteratorWithLock.java, OptimisticLock.java.

    üîÑ Read‚ÄëAfter‚ÄëWrite Semantics

    Two layers provide immediate visibility of recent writes:

    • Index‚Äëlevel UniqueCache (pre‚Äëflush) is checked first by Index.get and overlaid on iterators.
    • Segment delta cache (post‚Äëflush) is kept in memory when loaded; writes to a new delta file also update the in‚Äëmemory delta cache when present.

    Deletes are represented as tombstones by the value type descriptor. The read path treats a tombstone as ‚Äúnot found‚Äù.

    üßÆ Complexity and I/O Characteristics

    • Index‚Äëlevel cache probe: O(1) hash map
    • Segment delta cache probe: O(1) hash map
    • Bloom filter probe: O(k) where k is number of hash functions; no I/O
    • Sparse index probe: in‚Äëmemory list search over a small sample set (fast, cache‚Äëfriendly)
    • Local scan: sequential read within one chunk window of up to maxNumberOfKeysInIndexPage entries
    • Iterators: sequential over chunks; minimal seeks due to chunked layout

    These choices keep random access bounded and predictable, with sequential I/O for scans.

    ‚öôÔ∏è Configuration Knobs Affecting Reads

    • maxNumberOfKeysInSegmentChunk ‚Äî upper bound of keys per chunk; also the window size for a local scan from the sparse index pointer
    • Bloom filter parameters ‚Äî numberOfHashFunctions, indexSizeInBytes, falsePositiveProbability
    • diskIoBufferSize ‚Äî affects chunk and data block I/O buffering
    • Encoding/decoding filters ‚Äî enable CRC32, magic number and optional Snappy compression on read/write paths

    See: sst/IndexConfiguration and segment/SegmentConf.

    üõ°Ô∏è Integrity on the Read Path

    Decoding applies the inverse of the write pipeline when reading chunks:

    • Validate magic number
    • Verify CRC32
    • Decompress (if Snappy was enabled)

    Errors surface as exceptions; partial reads do not corrupt state.

    Key classes: chunkstore/ChunkStoreReaderImpl, chunkstore/ChunkFilterMagicNumberValidation, chunkstore/ChunkFilterCrc32Validation, chunkstore/ChunkFilterSnappyDecompress.
