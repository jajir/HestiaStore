    üíæ On-Disk Layout & File Names

    This page documents the files HestiaStore writes into an index directory, their naming conventions, how they evolve over time, and the atomic commit pattern used to keep them consistent.

    üìÇ Directory Layout (One Index per Directory)

    Top-level files: - index.map ‚Äî Global key‚Üísegment map (max key per segment). Sorted key‚ÜíSegmentId pairs. Updated atomically. - wal-00000.log, wal-00001.log, ‚Ä¶ ‚Äî Optional context logs if enabled. Useful for observability; not a recovery WAL.

    Per‚Äësegment files for segment segment-00000: - segment-00000.index ‚Äî Main SST in chunked format (ChunkStoreFile). Holds sorted key/value entries in chunks. - segment-00000.scarce ‚Äî Sparse index (key‚Üíchunk start position) to accelerate probes into .index. - segment-00000.bloom-filter ‚Äî Bloom filter backing store for negative lookups. - segment-00000.properties ‚Äî Segment properties (counts, delta numbering). - segment-00000.cache ‚Äî Optional seed file for the delta cache overlay (may be absent). - segment-00000-delta-000.cache, segment-00000-delta-001.cache, ‚Ä¶ ‚Äî Per‚Äësegment delta cache files created between compactions.

    Notes: - Segment ids are zero‚Äëbased and padded: segment-00000, segment-00001, ‚Ä¶ - Delta file counters are padded to 3 digits.

    üè∑Ô∏è Naming and Extensions

    • Main data: .index (chunked SST)
    • Sparse index: .scarce (sorted key‚Üíint pointer)
    • Bloom: .bloom-filter
    • Segment metadata: .properties
    • Delta/overlay: .cache (both seed cache and delta files)
    • Key‚Üísegment map: index.map
    • Context log: wal-xxxxx.log

    Code: segment/SegmentFiles.java, sst/KeySegmentCache.java, log/LogFileNamesManager.java.

    üß® Atomic Commit Pattern (*.tmp + rename)

    All persistent writers follow the same pattern: 1) openWriter() returns a writer bound to a temporary file (usually *.tmp). 2) Close the writer to flush OS buffers. 3) commit() atomically renames the temp file to its final name.

    Implications: - A crash never exposes a partially written visible file. At restart, either the old file or the new file is present. - Readers treat missing files as empty where applicable (e.g., no delta files ‚áí empty overlay).

    Code pointers: - Delta cache: sorteddatafile/SortedDataFileWriterTx (used by SegmentDeltaCacheWriter) - Main SST: chunkentryfile/ChunkEntryFileWriterTx ‚Üí chunkstore/ChunkStoreWriterTx ‚Üí datablockfile/DataBlockWriterTx - Sparse index: scarceindex/ScarceIndexWriterTx - Bloom filter: bloomfilter/BloomFilterWriterTx - Unsorted log: unsorteddatafile/UnsortedDataFileWriterTx

    üîÑ Segment Lifecycle

    1) New writes accumulate in the index write buffer; on flush they are routed by key into per‚Äësegment delta files segment-xxxxx-delta-YYY.cache. 2) Reads consult delta cache first, then .bloom-filter and .scarce to bound the probe into .index. 3) Compaction rewrites .index, .scarce, and .bloom-filter transactionally; on success, delta files are deleted and the in‚Äëmemory delta cache is cleared. 4) When a segment grows beyond the threshold, it is split: a new segment-xxxxx appears and index.map is updated atomically.

    üß¨ Chunked SST Anatomy

    The .index file is a sequence of fixed‚Äëcell chunks stored in a data‚Äëblock file. Each chunk has: - Header: magic number, version, payload length, CRC32, flags - Payload: a batch of sorted entries, optionally transformed by filters

    Filters add robustness and optional compression/obfuscation; their flags and order are recorded so the reader can invert them. See ‚ÄúFilters & Integrity‚Äù.

    Code: chunkstore/*, chunkentryfile/*.

    üîÅ Compatibility

    • Header fields (magic, version) allow future readers to validate format.
    • Sparse index and Bloom filter are rebuilt during compaction; no upgrade step is required beyond re‚Äëwriting segments if formats change in the future.
