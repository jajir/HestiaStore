    üõü Consistency & Recovery

    This page explains HestiaStore‚Äôs crash safety model and commit semantics. There is no WAL‚Äëbased crash recovery or multi‚Äëoperation transactions. Durability is driven by explicit flushes and by the fact that all data files are written via temporary files and atomically renamed on commit.

    üìú Scope and Guarantees

    • No automatic recovery: the system does not replay a WAL or roll back partial groups of operations after a crash.
    • Durability boundary: calling flush() or closing the index persists all writes that happened before the call. Writes that are only in memory (index‚Äëlevel buffer) and not flushed are not durable.
    • Atomic file replacement: data files are written to *.tmp and made visible via rename only after the writer is closed and the transaction is committed. A crash cannot produce partially written visible files.

    üíæ Where Writes Become Durable

    • Index‚Äëlevel buffer ‚Üí disk: SegmentIndex.flush() drains the in‚Äëmemory unique buffer into segment delta cache files. On close, the index also flushes.
    • Segment merge/compaction: when a segment compacts, the new main SST, sparse index, and Bloom filter are built via transactional writers; on commit they atomically replace the old ones.
    • Key‚Üísegment map (index.map): persisted via a transactional sorted data writer during flush or when updated.

    Relevant code: segmentindex/SegmentIndexImpl#flush(), segmentindex/CompactSupport, segmentindex/KeySegmentCache#optionalyFlush().

    ‚úçÔ∏è Transactional Write Primitives

    All main data files follow the same pattern: write to a temporary file, then atomically rename on commit().

    • Guarded transactions: GuardedWriteTransaction requires the resource to be closed before commit() and prevents double‚Äëcommit.
    • Single‚Äëcall helper: WriteTransaction.execute(writer -> { ‚Ä¶ }) does open ‚Üí write ‚Üí close ‚Üí commit.

    Key classes: - unsorteddatafile/UnsortedDataFileWriterTx ‚Üí rename(temp, final) on commit - sorteddatafile/SortedDataFileWriterTx ‚Üí rename(temp, final) on commit - datablockfile/DataBlockWriterTx ‚Üí used by chunk store writers - chunkstore/ChunkStoreWriterTx and chunkentryfile/ChunkEntryFileWriterTx ‚Üí layered over DataBlockWriterTx - bloomfilter/BloomFilterWriterTx ‚Üí writes new filter and swaps it in on commit

    üóÇÔ∏è File Types and Commit Paths

    • Segment delta cache files
    • Writer: segment/SegmentDeltaCacheWriter
    • Mechanism: SortedDataFileWriterTx.execute(‚Ä¶)

    • Naming: property counter assigns segmentId-delta-XXX.cache before write; if a crash happens before commit, the reader treats missing files as empty, so boot remains safe.

    • Main SST (chunked) + sparse index ("scarce index")

    • Writers: segment/SegmentFullWriterTx and segment/SegmentFullWriter
    • Internals: ChunkEntryFileWriterTx for SST, ScarceIndexWriterTx for the sparse index

    • Bloom filter: BloomFilterWriterTx builds a new filter and commits (rename) before the SST and sparse index are committed. This ordering avoids false negatives on restart.

    • Bloom filter

    • Writes to a temporary file via BloomFilterWriterTx.open() and commits with rename; also updates the in‚Äëmemory hash snapshot on commit.

    • Key‚Üísegment map (index.map)

    • Writer: SortedDataFileWriterTx.execute(‚Ä¶) inside KeySegmentCache.optionalyFlush()
    • Ensures the map is replaced atomically.

    üö´ What Is Not Transactional

    • Segment properties (counts and delta‚Äëfile numbering) are persisted via an overwrite (Directory.Access.OVERWRITE). They are updated after data files are committed, and are not critical to data correctness. If a crash corrupts or desynchronizes this metadata, the reader logic remains safe (e.g., missing delta file names yield empty reads) and you can re‚Äëestablish consistency via the checker below.

    Code: properties/PropertyStoreimpl and SegmentPropertiesManager.

    üí• Failure Model (Examples)

    • Crash while writing a delta file before commit: only *.tmp exists; it is ignored on boot; prior state remains valid.
    • Crash after committing a Bloom filter but before committing SST/sparse index: Bloom filter is ahead of data, which is safe (may increase positives but never produce false negatives).
    • Crash after committing SST/sparse index but before properties update: data is fully committed; metadata may lag but does not affect correctness.

    üß∞ Consistency Check and Repair

    • Run SegmentIndex.checkAndRepairConsistency() after an unexpected shutdown to verify that segments are well‚Äëformed and sorted and that the key‚Üísegment map is coherent. This walks all segments, checks ordering and basic invariants, and raises an error if it finds non‚Äërecoverable issues.

    Key classes: segmentindex/IndexConsistencyChecker, segment/SegmentConsistencyChecker.

    üë©‚Äçüíª Developer Notes: open()/commit() and *.tmp

    • open() returns a writer bound to a temporary file (typically with a .tmp suffix). You must close the writer before calling commit().
    • commit() performs an atomic rename(temp, final) so either the old file or the new file is visible on disk.
    • Prefer execute(writer -> {‚Ä¶}) to ensure the correct order: open ‚Üí write ‚Üí close ‚Üí commit.

    Examples in code: - sorteddatafile/SortedDataFileWriterTx#open() ‚Üí commit() renames temp to final - unsorteddatafile/UnsortedDataFileWriterTx#open() ‚Üí commit() renames temp to final - datablockfile/DataBlockWriterTx#open() ‚Üí commit() renames temp to final - bloomfilter/BloomFilterWriterTx#open() ‚Üí commit() renames temp to final and swaps hash

    üß≠ Practical Guidance

    • Call flush() on periodic boundaries and always before shutdown to persist in‚Äëmemory writes.
    • After a crash, reopen the index and run checkAndRepairConsistency(); optionally trigger a compact() to collapse delta caches.
    • Remember there is no WAL: durability is guaranteed at the flush()/close boundaries and via atomic file replacement for all data files.
